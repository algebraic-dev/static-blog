Pensando a Linguagem - MKE1

Recentemente vim tendo varios problemas relacionados a linguagens de programação e por isso, para expressar minha indignação (ou talvez só minha falta de habilidade com essas linguagens), eu decidi fazer alguns posts. Esse post é totalmente dedicado a meus problemas com PHP, JS e Haskell além de tentativas falhas de ter usado algumas outras linguagens estilo Unison que me fizeram refletir sobre como software grandes poderiam ser projetados para serem mais faceis de serem compreendidos e mais faceis de arrumar sem quebrar partes. Além de falar mal de linguagens e meus problemas com elas, eu irei fazer um "devlog" de como fazer uma linguagem de programação na linguagem Idris do 0.

Coisas que a linguagem NÃO vai ter:

Começando de maneira bem granular, vou citar alguns pontos que eu ODEIO pelo simples motivo desses pontos servirem de fundação e guia para todas as coisas que vamos ter na linguagem.

  • Variáveis globais: Essa importante feature para software moderno, que sem isso não teriamos a absurda performance e facilidade, faz com que haja problemas com a MINHA leitura do código. Nunca sei em tempo de compilação qual é o valor e se aquilo pode levar o programa a um estado inválido. Além de que podem haver problemas de concorrencia que só vão poder ser arrumados com Mutexes, RwLocks, Semaforos e etc.
  • Absurda quantidade de magia: Basicamente, mecanismos que são dados pela linguagem mas parte deles é feito de maneira magica que não tem como implementar em termos comuns. Poucas vezes isso é benefico como algumas built-in functions do Erlang como o recv e o send que mandam e recebem mensagens de green-threads que são construções da VM Beam. Só que em outros casos elas introduzem complicações na leitura e até coisas que podem variar, de infinitas maneiras, como o programa deve ser lido dependendo de um dado. Um grande exemplo disso é o extract do PHP que introduz uma variação nas varáveis que estão no escopo no momento e isso pode variar de infinitas maneiras como de acordo com um argumento de uma função e isso me lembra muito um comentário cheio de ódio sobre ADVPL.
  • Classes: Pessoas amam hierarquias, classes, design patterns mas eu gosto de ver a programação de uma maneira mais simples. Que tal se tudo for um caso de transformação entre dados e sim.. as vezes vamos ter que classificar coisas como Functors, Monoids e Monads mas essas são bases bem mais simples e mais fortes porque ao invés de necessitarmos de Design Patterns que falam o que um pedaço de código significa de acordo com um padrão, essas classificações de programaçao funcional, dão a habilidade de se utilizar certas funções para esse tipo já que ela se encaixa na categoria. Quando usamos POO com Design Patterns, o significado das coisas é mais abrangente e temos menos facilidade de classificar se certo pedaço de código contem uma cateracteristica.
  • Tipagem dinamica: Eu tenho problemas de atenção e memória, tenho um terrível costume de escrever 1000+ linhas de código em uma linguagem dinamica e não lembrar qual era o tipo e as limitações de certa varíavel que chegou depois de passar por umas 200 funções e 200 classes.

Features divertidas para gente testar

  • Efeitos algebricos: Todas funções que mexem com mundo real criam Side-Effects. Isso não é algo que eu quero, como eu disse no primeiro tópico, minha memória é curta e eu não curto não saber o que ta acontecendo em certo pedaço de código. Então para evitar isso, podemos explicitamente colocar como que vamos modificar o mundo real por exemplo:
    main : { IO } ()
    main = 
          print "Hi"
    Parece meio confuso de primeira vista, mas é bem simples o significado, então vamos demembrar isso linha por linha:
    1. Definimos o tipo de main sendo exatamente { IO } (). Vamos começar pelo (), essa parte significa o tipo Unit que tem exatamente um construtor, o proprio () ou seja () é do tipo (). Mas ué pra que isso serve? Simples: falar que essa função retorna um valor que normalmente não vamos usar já que essa linguagem NÃO TEM NULL. Já o { IO } Define que a função irá produzir um efeito de Input/Output ou seja, se comunicar com o mundo real (printar, pegar linha do terminal, produzir som, luz, agua ou fogo).
    2. Agora podemos definir o que essa função faz e o que ela é faz é basicamente printar "Hi" na tela. Print é uma função que recebe uma String, produz um efeito de IO e também retorna () ou seja String -> { IO } ()
